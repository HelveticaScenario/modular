# WAV Sample Loading (`$wavs()`) Design

## Goal

Add a `$wavs()` DSL function that loads WAV files from a `wavs/` folder in the workspace, with autogenerated TypeScript types matching the folder structure, cached loading, and a `$sampler` module for triggered playback.

## Architecture Overview

WAV files are loaded eagerly on the main thread when accessed via a nested Proxy during DSL execution. Loaded data is cached in Rust behind `Arc<WavData>`. The `Wav` param type follows the same two-phase pattern as `Buffer`: deserialize a lightweight path identifier, then resolve `Arc<WavData>` during `connect()` on the audio thread. A filesystem watcher keeps Monaco types in sync with the `wavs/` folder.

## Data Types

### `SampleBuffer` (new, `crates/modular_core/src/types.rs`)

Shared audio sample storage extracted from `BufferData`. Plain struct, no `UnsafeCell`:

```rust
pub struct SampleBuffer {
    channels: usize,
    frame_count: usize,
    samples: Vec<Vec<f32>>,
}
```

**Read methods** (`&self`): `read(channel, frame) -> f32`, `channel_count()`, `frame_count()`, `with_data(closure)`, `snapshot() -> Vec<Vec<f32>>`.

**Write methods** (`&mut self`): `write(channel, frame, value)`, `fill(value)`, `with_data_mut(closure)`.

**Constructors**: `new_zeroed(channels, frame_count)`, `from_samples(Vec<Vec<f32>>)`.

### `BufferData` refactored (same file)

Wraps `SampleBuffer` in `UnsafeCell` for interior mutability on the audio thread:

```rust
pub struct BufferData {
    buffer: UnsafeCell<SampleBuffer>,
    write_index: UnsafeCell<usize>,
}
```

All existing `BufferData` methods are preserved — they delegate to `SampleBuffer` through `UnsafeCell`. `unsafe impl Send + Sync` remains. `copy_overlap_from` stays on `BufferData`. The `Buffer` param type and all existing modules (`$buffer`, `$bufRead`, `$delayRead`) are unchanged in their public API.

### `WavData` (new, same file)

Immutable loaded sample data. No `UnsafeCell`:

```rust
pub struct WavData {
    buffer: SampleBuffer,
    sample_rate: f32, // original sample rate before resampling
}
```

Exposes only read methods from `SampleBuffer`. No write access. Naturally `Send + Sync` because it contains no interior mutability. Wrapped in `Arc<WavData>` for shared ownership across cache and audio thread.

### `Wav` param type (new, same file)

Connection handle for WAV data, following the `Buffer` pattern:

```rust
pub struct Wav {
    path: String,
    channels: usize,
    #[serde(skip)] #[deserr(skip)] #[schemars(skip)]
    cached_data: Option<Arc<WavData>>,
}
```

**Deserialization (`Deserr` impl):** Parses `path` (string) and `channels` (usize) from JSON. `cached_data` starts as `None`.

**Connection (`Connect` impl):** `connect(&mut self, patch: &Patch)` looks up `self.path` in `patch.wav_data: HashMap<String, Arc<WavData>>` and clones the `Arc` into `cached_data`.

**Read access:** `channel_count()`, `frame_count()`, `read(channel, frame) -> f32`, `is_loaded() -> bool`.

## WAV Loading and Cache

### Cache location

A `WavCache` struct on the `Synthesizer` (N-API layer, `crates/modular/src/lib.rs`):

```rust
struct WavCacheEntry {
    data: Arc<WavData>,
    mtime: SystemTime,
}

struct WavCache {
    entries: HashMap<String, WavCacheEntry>,
    workspace_path: PathBuf,
}
```

### N-API function: `load_wav`

```rust
#[napi]
pub fn load_wav(&mut self, path: String) -> Result<WavLoadResult> { ... }
```

Called synchronously from JS during DSL execution when a `$wavs()` property is accessed.

**Steps:**

1. Resolve path: `workspace_path/wavs/{path}.wav`
2. Check cache: if entry exists and mtime matches, return cached info
3. Cache miss or stale: read file with `hound::WavReader`, decode to `Vec<Vec<f32>>`
4. Resample to engine sample rate if needed (linear interpolation)
5. Wrap in `Arc::new(WavData { buffer: SampleBuffer::from_samples(samples), sample_rate })`
6. Store in cache, return `WavLoadResult { channels, frame_count, path }`

### Patch integration

The `Patch` struct gets a new field: `wav_data: HashMap<String, Arc<WavData>>`.

The `PatchUpdate` struct (incremental diff sent to audio thread via command queue) also gets a `wav_data: HashMap<String, Arc<WavData>>` field. Before building the update on the main thread, the N-API layer clones the entire `WavCache` entries map into this field. This is cheap — it's just cloning `Arc`s.

During `apply_patch_update` on the audio thread, the old `wav_data` map on the `Patch` is swapped out for the new one. The old map is sent to the garbage channel for main-thread drop. The `Arc`s in the old map just decrement their refcount — no actual data is freed because the `WavCache` still holds references. This is a cheap operation: swapping a `HashMap` of `Arc`s involves no audio data copying.

After the swap, `connect()` runs on all modules. `Wav::connect()` looks up its path in `patch.wav_data` and caches the `Arc<WavData>`.

## DSL Integration

### `$wavs()` function (`src/main/dsl/executor.ts`)

Returns a deeply nested `Proxy` mirroring the `wavs/` folder structure.

**Property access behavior:**

- Directory node → returns another nested Proxy for deeper traversal
- File leaf (`.wav`) → calls `synthesizer.loadWav(path)` synchronously via N-API, returns `{ type: 'wav_ref', path, channels, frameCount }`
- Missing path → throws descriptive error immediately

**Example:**

```js
$wavs().kick; // loads wavs/kick.wav, returns wav_ref handle
$wavs().tables.boom; // loads wavs/tables/boom.wav, returns wav_ref handle
```

The folder tree structure is provided to the executor from the main process (scanned on startup, updated by watcher).

### `$wavs()` type generation (`src/main/dsl/typescriptLibGen.ts`)

A function generates a TypeScript interface tree from the `wavs/` folder scan:

```typescript
interface WavHandle {
    readonly type: 'wav_ref';
    readonly path: string;
    readonly channels: number;
    readonly frameCount: number;
}

interface WavsTree {
    readonly kick: WavHandle;
    readonly tables: {
        readonly boom: WavHandle;
    };
}

function $wavs(): WavsTree;
```

This is appended to the output of `buildLibSource()`.

## Filesystem Watching

### Watcher (`src/main/main.ts`)

On startup, scan the workspace for a `wavs/` directory. Use `fs.watch` (recursive) to detect changes (file added/removed/renamed).

**On change:**

1. Rescan `wavs/` folder tree
2. Regenerate `$wavs` type declaration
3. Invalidate `cachedLibSource`
4. Push updated types to renderer via `webContents.send()` (follow `CONFIG_ON_CHANGE` pattern)
5. Renderer reloads Monaco extra lib

The watcher does NOT trigger WAV loading — loading only happens during DSL execution when a file is accessed. The watcher only updates types for autocomplete.

## `$sampler` Module

### Location

New module category: `crates/modular_core/src/dsp/samplers/sampler.rs`

### Params

```rust
#[derive(Clone, Deserr, JsonSchema, Connect, ChannelCount, SignalParams)]
#[deserr(rename_all = camelCase, deny_unknown_fields)]
struct SamplerParams {
    wav: Wav,                    // positional arg: the loaded WAV data
    #[deserr(default)]
    gate: MonoSignal,            // trigger input — rising edge starts playback
    #[deserr(default = 1.0)]
    speed: MonoSignal,           // playback rate (1.0 = normal, 2.0 = double, negative = reverse)
}
```

### Outputs

Single polyphonic output matching WAV channel count.

### Behavior (oneshot mode)

- Gate rising edge (Schmitt trigger) resets playback position to 0 and starts playback
- Position advances by `speed` samples per tick
- When position reaches end of sample (or beginning, if reverse), output silence
- Uses Hermite interpolation from `SampleBuffer` for sub-sample accuracy
- Channel count derived from `wav.channel_count()` at deserialization time

### State (on params struct, `#[serde(skip)]` `#[deserr(skip)]` `#[schemars(skip)]`)

```rust
#[serde(skip)] #[deserr(skip)] #[schemars(skip)]
position: f64,           // current fractional playback position
#[serde(skip)] #[deserr(skip)] #[schemars(skip)]
playing: bool,           // whether currently producing output
#[serde(skip)] #[deserr(skip)] #[schemars(skip)]
gate_trigger: SchmittTrigger,  // edge detection
```

## Error Handling

- **Missing file:** `$wavs().nonexistent` throws immediately in JS with a clear error message including the expected path
- **Invalid WAV file:** `load_wav` returns an N-API error, which propagates as a JS exception during DSL execution
- **WAV decode errors:** Same — reported as DSL execution errors with file path context
- **Empty wavs/ folder:** `$wavs()` returns an empty proxy, type is `{}` — no errors unless a property is accessed

## Testing Strategy

- **Rust unit tests** (`cargo test -p modular_core`): `SampleBuffer` read/write, `WavData` construction, `Wav` param deserialization and connect, `$sampler` module process logic
- **N-API integration** (`yarn test:unit`): `load_wav` with test WAV files, cache hit/miss behavior, mtime invalidation
- **DSL integration** (`yarn test:unit`): `$wavs()` proxy behavior, type generation, `$sampler` factory
- **E2E** (`yarn test:e2e`): Full roundtrip — WAV file in `wavs/`, DSL patch using `$sampler($wavs().file)`, audio output
