# Sampleable Sync Removal Design

## Overview

Finish the send-only runtime cleanup by making `Sampleable` explicitly `Send` but not `Sync`. Published modules are transferred across threads once, then owned and executed only by the audio thread. The type model should match that rule directly.

## Goals

- Remove `Sync` from the `Sampleable` trait
- Remove generated `Sync` claims and any manual runtime `Sync` claims that exist only to support fake shared module access
- Preserve current owned-box runtime behavior, patch updates, state transfer, and reconnect semantics
- Keep truly shared data types `Sync` where they are actually shared across threads

## Non-Goals

- Refactoring unrelated generic `Send + Sync` APIs unless compilation proves they are coupled to `Sampleable`
- Changing DSP behavior, patch semantics, or module connection behavior
- Reworking the pattern system just because it also uses `Send + Sync` bounds

## Selected Approach

Use runtime-only `Sync` removal plus compile-driven adjacent cleanup.

This means:

- Change `Sampleable` from `MessageHandler + Send + Sync` to `MessageHandler + Send`
- Remove generated `unsafe impl Sync` from module wrappers in `crates/modular_derive/src/module_attr.rs`
- Remove manual `unsafe impl Sync` from runtime helper types whose only justification was the old fake shared-module model
- Keep `Sync` for true shared data types such as `BufferData`, `WavData`, and `Arc`-backed assets that may legitimately have concurrent owners
- Use compiler errors to identify any broader cleanup that is genuinely required by this trait change
- If removing `Signal: Sync` reaches the pattern system through `SeqPatternParam -> Pattern<SeqValue> -> SeqValue::Signal`, stop there and treat that broader generic cleanup as out of scope for this slice

## Files In Scope

- `crates/modular_core/src/types.rs`
- `crates/modular_derive/src/module_attr.rs`
- Tests that explicitly or implicitly assert old `Sampleable: Sync` behavior
- Any direct fallout sites that stop compiling because they still assume `dyn Sampleable: Sync`

## Safety Model

After this change, the runtime contract is:

1. The main thread may construct unpublished modules.
2. Modules may cross to the audio thread by `Send` ownership transfer.
3. Once published, module methods are not concurrently callable from multiple threads.
4. Graph mutation and audio processing remain phase-separated.

Removing `Sync` makes that contract explicit. It does not change the underlying execution model.

## Expected Fallout

The most likely direct fallout is:

- Trait-object uses that still require `dyn Sampleable: Sync`
- Wrapper code generated by the proc macro
- Manual `unsafe impl Sync` on connection-cache helper types that should now be send-only runtime objects

The pattern system and other generic `Send + Sync` code are only in scope if the compiler proves they directly depend on `Sampleable` remaining `Sync`.

Actual coupling discovered during implementation: `Signal` flows into `SeqValue`, `SeqPatternParam` stores `Option<Pattern<SeqValue>>`, and `Pattern<T>` still requires `T: Clone + Send + Sync + 'static` because its query closure is stored as `Arc<dyn Fn(&State) -> Vec<Hap<T>> + Send + Sync>`. That keeps `Signal: Sync` in place for now even though `Buffer` can be made send-only.

## Testing Strategy

Use TDD for the transition:

1. Remove the bound and wrapper `Sync` impls to produce the real RED compiler/test failures.
2. Fix only the directly related fallout.
3. Re-run the send-only verification slice:
   - `cargo test -p modular_core`
   - `cargo test -p modular test_single_module_update_re_registers_message_listeners -- --nocapture`
   - `cargo test -p modular test_patch_update_remap_re_registers_message_listeners -- --nocapture`
   - `cargo test -p modular --no-run`

Add or adjust focused tests only where needed to prove the new contract or protect a discovered regression.

## Notes

- This spec intentionally covers only the final `Sync`-removal slice, not the broader raw-pointer or owned-box work already completed.
- Implemented end state for this slice: `Sampleable` is `Send` and not `Sync`, generated wrappers no longer claim `Send` or `Sync`, `Buffer` is `!Sync`, and `Signal` remains `Sync` until the pattern-system bounds are relaxed in a separate change.
- Per prior instruction in this session, this spec is written without creating a git commit.
