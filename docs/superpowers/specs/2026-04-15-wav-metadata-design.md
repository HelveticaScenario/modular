# WAV Metadata Extraction for `$wavs()` System

**Date:** 2026-04-15
**Status:** Approved

## Summary

Extend the `$wavs()` DSL system to expose WAV file metadata on the wav handle. Currently handles only expose `{ type, path, channels }`. This adds format info (sample rate, frame count, duration, bit depth), sampler chunk data (pitch, loops), cue points, and ACID chunk data (BPM, beats, time signature, playback mode).

## Approach

Use the `riff` crate (pure Rust, zero dependencies, 11M+ downloads) for RIFF chunk traversal. Parse `fmt `, `smpl`, `cue `/`adtl`, and `acid` chunk binary data manually. Keep `hound` for audio sample decoding — the `riff` crate only handles metadata extraction.

## WavHandle Shape

```typescript
type WavHandle = {
    readonly type: 'wav_ref';
    readonly path: string;
    readonly channels: number;
    readonly sampleRate: number;
    readonly frameCount: number;
    readonly duration: number; // seconds (frameCount / original sampleRate)
    readonly bitDepth: number;
    readonly pitch?: number; // V/Oct (0V = C4). From smpl chunk (precise, with fine tune) or acid rootNote (fallback)
    readonly playback?: 'one-shot' | 'loop'; // acid chunk
    readonly bpm?: number; // acid chunk
    readonly beats?: number; // acid chunk
    readonly timeSignature?: {
        // acid chunk
        readonly num: number;
        readonly den: number;
    };
    readonly loops: ReadonlyArray<{
        // smpl chunk, empty array if absent
        readonly type: 'forward' | 'pingpong' | 'backward';
        readonly start: number; // seconds (f64)
        readonly end: number; // seconds (f64)
    }>;
    readonly cuePoints: ReadonlyArray<{
        // cue chunk, empty array if absent
        readonly position: number; // seconds (f64)
        readonly label: string; // from adtl/labl, or empty string
    }>;
};
```

### Precision Note

Loop start/end and cue point positions are stored as f64 seconds. This provides exact round-trip precision to sample offsets for any realistic file length. However, when these values are used as f32 signals in the DSP engine, precision degrades for long files (e.g. at 30 minutes / 48kHz, f32 has ~10 samples of error). This is acceptable for the initial implementation but worth noting for future work.

### Pitch Derivation

V/Oct from `smpl` chunk: `(midi_unity_note - 60 + midi_pitch_fraction_cents / 100) / 12`

Where `midi_pitch_fraction_cents` converts the raw 32-bit fraction to cents (0-99.99...).

If no `smpl` chunk but `acid` chunk has a root note: `(acid_root_note - 60) / 12` (no fine tune available).

## Architecture

### New Files

- **`crates/modular/src/wav_metadata.rs`** — Metadata extraction using `riff` crate. Parses `fmt `, `smpl`, `cue `, `adtl`/`labl`, and `acid` chunks. Returns a `WavMetadata` struct.

### Modified Files

- **`crates/modular/Cargo.toml`** — Add `riff = "2.0"` dependency.
- **`crates/modular/src/lib.rs`** — Expand `WavLoadInfo` N-API struct with all metadata fields. Call `wav_metadata::extract()` in `WavCache::load()`. Convert sample offsets to seconds using the file's original sample rate.
- **`src/main/dsl/executor.ts`** — Expand the wav handle object returned by `makeProxy()` to include all new fields from `WavLoadInfo`.
- **`src/main/dsl/typescriptLibGen.ts`** — Update `WavHandle` type definition for Monaco autocomplete.
- **`crates/modular/index.d.ts`** — Regenerated by `yarn generate-lib` after N-API changes.

### Data Flow

1. `WavCache::load()` opens the file with `riff::Chunk::read()`, iterates chunks, calls `wav_metadata::extract()`
2. `extract()` returns `WavMetadata` with all parsed fields
3. `WavLoadInfo` N-API struct carries metadata to TypeScript
4. `executor.ts` builds the wav handle object with all fields
5. `typescriptLibGen.ts` provides Monaco types for autocomplete

### Chunk Formats Parsed

#### `smpl` chunk (sampler)

- Header: manufacturer, product, sample_period, midi_unity_note (u32), midi_pitch_fraction (u32), smpte_format, smpte_offset, num_sample_loops, sampler_data_size
- Per loop: cue_point_id (u32), type (u32: 0=forward, 1=pingpong, 2=backward), start (u32 sample offset), end (u32 sample offset), fraction (u32), play_count (u32)

#### `cue ` chunk (cue points)

- Header: num_cue_points (u32)
- Per cue: id (u32), position (u32), data_chunk_id (fourcc), chunk_start (u32), block_start (u32), sample_offset (u32)

#### `adtl`/`labl` (labels for cue points)

- LIST chunk with type `adtl`, containing `labl` sub-chunks
- Per labl: cue_point_id (u32), text (null-terminated string)

#### `acid` chunk

- Flags (u32): bit 0 = one-shot, bit 1 = root note valid, bit 3 = disk-based, bit 4 = ?? (often set)
- Root note (u16): MIDI note number
- Unknown (u16)
- Unknown (f32)
- Beats (u32)
- Time signature: numerator (u16), denominator (u16) — denominator as power of 2
- Tempo (f32): BPM

## Testing

### Rust unit tests (`crates/modular/src/wav_metadata.rs`)

- Parse a minimal WAV with only `fmt ` → returns format info, empty loops/cues/acid
- Parse WAV with `smpl` chunk → correct pitch, loops
- Parse WAV with `cue ` + `adtl`/`labl` → correct positions and labels
- Parse WAV with `acid` chunk → correct BPM, beats, time sig, playback mode
- Pitch derivation: MIDI 60 → 0V, MIDI 72 → 1V, MIDI 69 + fine tune → correct V/Oct
- Missing chunks → optional fields are None, arrays empty

### TypeScript unit tests (`src/main/dsl/__tests__/executor.test.ts`)

- Wav handle exposes all metadata fields from loadWav
- Monaco type generation includes new WavHandle fields
